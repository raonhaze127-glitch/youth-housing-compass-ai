# 통합 개발 백로그

## 범위

기관별 공고 수집·분석 기능 중 입지 분석을 제외한 기능을 청년주거나침반에 단계적으로 도입합니다. 각 단계는 앞 단계의 데이터와 검증 결과에 의존합니다.

## 0단계: 도입 기반 확정

### 작업

- 참고 수집 구조와 코드 사용 권한 확인
- 재사용 파일과 청나주 신규 코드의 출처 기록 방식 결정
- 공고, 사용자 프로필, 자격 판정 공통 스키마 확정
- 실제값·추정값·확인 필요 상태 정의
- Python 서비스 배치 위치와 배포 환경 결정
- API 계약과 오류 응답 형식 문서화

### 완료 조건

- 코드 이식 범위와 출처가 기록돼 있습니다.
- Next.js와 수집 API 사이의 요청·응답 예제가 승인돼 있습니다.
- 샘플 데이터와 실데이터가 구분됩니다.

## 1단계: 실공고 수집과 조회

진행 상태: 독립 API, 공통 모델, 샘플 소스와 LH·SH·GH 공공주택 및 청약홈 APT 국민주택 직접 수집기, FastAPI 실행 환경과 Next.js 연결까지 구현했습니다. 공공데이터 키가 없으면 SH·GH 부분 수집으로 동작합니다. 청약홈 민영 및 나머지 4개 채널은 향후 민간주택 확장용으로 보존하되 현재 기본 범위에서는 비활성화합니다. GitHub Actions가 마지막 검증 실공고를 정적 JSON으로 보존해 Render 초기화 시에도 추천을 유지합니다.

### 작업

- Python/FastAPI 공고 서비스 골격 추가
- 청약홈 APT 국민주택 수집과 민영주택 차단
- LH API 연동
- SH·GH 게시판 수집기 연동
- 기관별 응답을 청나주 공통 스키마로 정규화
- 접수중, 모집예정, 마감 상태 계산
- 수집 실패, 오래된 캐시, 일정 미확인 상태 표시
- Next.js 추천 화면을 실공고 API와 연결할 수 있는 어댑터 추가
- 검증된 실공고 스냅샷 생성과 API 장애 폴백

### 완료 조건

- 기관별 수집 결과가 하나의 공고 목록으로 조회됩니다.
- 기관 장애가 전체 조회 장애로 번지지 않습니다.
- 원문 링크와 수집 시각이 모든 공고에 포함됩니다.
- 샘플 모드와 실데이터 모드를 명확히 구분할 수 있습니다.

## 2단계: 원문 분석과 변동 추적

진행 상태: HTML·PDF·HWPX 원문 추출을 내부 구현했습니다. 공고 목록은 SQLite에 누적하고 내용 해시 기반 신규·수정 이력과 동기화 상태를 저장해 변동 패널에서 사용합니다. 바이너리 HWP 추출, 첨부파일 버전 저장, Render 재배포에도 유지되는 외부 영속 DB는 남아 있습니다.

### 작업

- HTML, PDF, HWP/HWPX 원문 추출기 도입
- 자격, 공급대상, 일정, 금액, 준비서류, 유의사항 섹션 분리
- 원문과 추출 결과의 버전·내용 해시 저장
- 일일 스냅샷 또는 DB 버전 비교
- 신규, 수정, 마감, 삭제 이벤트 기록
- 정정공고와 첨부파일 교체 감지
- 수집기·추출기 회귀 테스트 구축

### 완료 조건

- 지원 기관의 공고 원문을 재처리할 수 있습니다.
- 변경 전후 필드와 원문 버전을 확인할 수 있습니다.
- 추출 실패가 숨겨지지 않고 확인 필요 상태로 노출됩니다.

## 3단계: 프로필과 자격·가점 엔진

진행 상태: SQLite 프로필 저장, 청약 가점·1순위·특별공급 API, 실공고 적합도를 연결했습니다. 청년주거 사업별 세부 규칙 버전 관리는 남아 있습니다.

### 작업

- 계정별 사용자 프로필 저장
- 청약 가점 84점 계산과 테스트 이식
- 1순위와 특별공급 사전 점검 도입
- 행복주택, 청년매입임대, 전세임대, 월세지원 규칙 모듈 추가
- 충족, 불충족, 확인 필요 3상태 판정
- 규칙 출처, 적용일, 버전 저장
- 사용자 조건이 부족할 때 필요한 질문 생성

### 완료 조건

- 같은 입력과 규칙 버전은 같은 판정 결과를 만듭니다.
- 판정마다 근거와 확인할 원문 항목이 표시됩니다.
- 실제 신청 자격 확정이 아니라는 안내가 화면에 표시됩니다.

## 4단계: 추천과 대화형 공고 해석

진행 상태: 이전 추천 번호 기반 후속 질문은 구현했습니다. 원문 자격 섹션 추출은 시범 API만 남기고 품질 보강 전까지 카드에서 숨깁니다. 외부 LLM과 검색 인덱스를 사용하는 RAG는 남아 있습니다.

### 작업

- 모집 상태, 자격 결과, 지역, 관심 유형을 결합한 추천 정렬
- 원문 청크와 공고 메타데이터 검색 인덱스 구축
- 질문과 관련된 원문 근거 검색
- 공고별 자격, 일정, 금액, 준비서류 질의응답
- 답변에 공고명, 원문 URL, 기준 날짜, 근거 섹션 표시
- 근거가 부족하면 모른다고 응답하고 원문 확인 유도
- 대화 이력과 명시적 피드백 저장

### 완료 조건

- 답변이 저장된 공고 원문과 계산 결과를 근거로 합니다.
- 서로 다른 공고의 조건을 섞지 않습니다.
- 사용자 피드백은 자동 학습이 아니라 평가·개선 데이터로 분리 저장됩니다.

## 5단계: 경쟁률·일정·즐겨찾기

진행 상태: ICS 다운로드와 브라우저 사용자별 관심 공고 저장을 구현했습니다. 경쟁률은 통계 추정 시범 API만 있어 카드에서 숨기고, LH·SH·GH 공식 모집결과 연동을 향후 작업으로 둡니다.

### 작업

- 실제 경쟁률 조회
- 지역 과거 이력과 통계 추정 폴백
- 실제값과 추정값을 화면에서 구분
- 접수, 당첨 발표, 계약 일정을 공고별로 관리
- ICS 생성과 다운로드
- 즐겨찾기와 관심 공고 목록
- 일정 변경 시 즐겨찾기 사용자 대상 이벤트 생성

### 완료 조건

- 경쟁률 출처와 정확도 수준이 표시됩니다.
- 추정 일정은 확정 일정처럼 표현되지 않습니다.
- 즐겨찾기 공고의 변경 이력을 확인할 수 있습니다.

## 6단계: 운영 안정화

진행 상태: Slack·Telegram 발송은 서비스 범위에서 제외했습니다. 공고 수집과 분석 상태를 안정적으로 운영하는 작업만 후속 범위로 둡니다.

### 작업

- 수집·분석 작업 모니터링
- API 호출량과 외부 사이트 변경 감지
- 개인정보 삭제 흐름

### 완료 조건

- 외부 공고 서버 장애가 샘플 시연 흐름을 막지 않습니다.
- 사용자가 프로필 데이터를 직접 삭제할 수 있습니다.

## 구현 우선순위

1. 실공고 수집과 공통 스키마
2. 원문 분석과 변동 추적
3. 청년주거 자격 규칙
4. 근거 기반 추천과 대화
5. 경쟁률·일정·즐겨찾기
6. 운영 안정화

공모전 이후 확장에서는 기능 수보다 공고 출처, 판정 근거, 갱신 시각을 신뢰할 수 있게 만드는 작업을 우선합니다.
