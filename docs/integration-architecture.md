# k-apt-alert 통합 아키텍처

## 목적

k-apt-alert의 공고 수집·가점 계산·원문 추출·경쟁률·일정·변동 추적 기능을 청년주거나침반의 사용자 경험과 결합합니다. 입지 분석과 Slack·Telegram 발송은 범위에서 제외합니다.

이 문서는 목표 구조와 현재 통합 경계를 함께 정의합니다. 청약홈 5종·LH 공개 API와 SH·GH HTML 직접 수집, 가점·원문·경쟁률·일정·변동 기능을 청나주 FastAPI 내부에 구현했습니다. 영속 원문·버전 저장소는 목표 구조로 남아 있습니다. 정확한 현재 동작 범위는 `project-overview.md`를 기준으로 합니다.

## 통합 원칙

1. 현재 Next.js 화면과 추천 MVP는 사용자 접점으로 유지합니다.
2. k-apt-alert는 기능 계약과 기관별 접근 경로를 확인하는 참고 구현으로만 활용하고 소스 코드는 복사하지 않습니다.
3. 외부 공고는 기관별 원본과 청나주 공통 스키마를 분리해 저장합니다.
4. 자격과 가점은 LLM이 임의 계산하지 않고 버전이 표시된 결정론 규칙으로 계산합니다.
5. LLM은 원문 근거를 검색하고 설명하는 역할을 맡으며 최종 자격을 확정하지 않습니다.
6. 실제값, 추정값, 사용자가 직접 확인해야 하는 값을 응답에서 구분합니다.
7. 입지 분석 및 지도 기반 주변 시설 평가는 구현하지 않습니다.

## 목표 구조

```text
청년주거나침반
├─ web (Next.js)
│  ├─ 사용자 입력과 대화
│  ├─ 프로필·즐겨찾기
│  ├─ 추천·자격·경쟁률 화면
│  └─ 일정 관리
│
├─ announcement-api (Python/FastAPI)
│  ├─ 기관별 공고 수집
│  ├─ 공통 스키마 정규화
│  ├─ 공고 원문·첨부 추출
│  ├─ 자격·가점 규칙 계산
│  ├─ 경쟁률 조회
│  └─ ICS 생성
│
├─ workers
│  ├─ 정기 수집
│  ├─ 공고 변동 비교
│  └─ 원문 재처리
│
└─ persistence
   ├─ PostgreSQL: 공고·프로필·판정·즐겨찾기
   ├─ 원문 저장소: HTML·PDF·HWP/HWPX
   └─ 검색 인덱스: 공고 원문 청크·근거 메타데이터
```

## 처리 흐름

1. 수집기가 청약홈, LH, SH, GH 등 기관별 데이터를 가져옵니다.
2. 원본 응답과 수집 시각을 보존합니다.
3. 정규화기가 기관별 필드를 청나주 공통 공고 스키마로 변환합니다.
4. 원문 추출기가 HTML과 첨부파일에서 자격, 일정, 금액, 공급대상, 유의사항을 분리합니다.
5. 변동 추적기가 이전 버전과 비교해 신규, 수정, 마감 또는 삭제를 기록합니다.
6. 규칙 엔진이 사용자 프로필과 공고를 비교해 충족, 불충족, 확인 필요를 반환합니다.
7. 추천 엔진이 모집 상태, 자격 결과, 관심 유형, 지역을 함께 고려해 정렬합니다.
8. 대화 계층은 저장된 원문 근거와 계산 결과를 이용해 설명합니다.

## 기능별 도입 판단

| 기능 | k-apt-alert 활용 | 청나주 작업 |
|---|---|---|
| 8개 채널 공고 수집 | 수집기와 캐시 전략 참고·이식 | 청년 공공임대·주거지원 분류 확장 |
| LH·SH·GH 수집 | 기관별 수집기 활용 | 실패 감지, 일정 보강, 데이터 검증 |
| 공고 정규화 | 공통 필드 변환 로직 활용 | 청나주 공통 스키마로 매핑 |
| 청약 가점 | 결정론 계산과 테스트 활용 | 규칙 버전과 근거 문구 저장 |
| 특별공급·1순위 | 초기 사전 점검으로 활용 | 최신 규정 검증과 확인 필요 상태 강화 |
| 청년주거 자격 | 직접 재사용 범위가 작음 | 행복주택·매입임대·월세지원별 규칙 신규 구축 |
| 공고 원문 추출 | HTML·PDF·HWP/HWPX 추출기 활용 | 원문 저장, 재처리, 품질 점수 추가 |
| 공고 해석·질의응답 | Skill의 응답 원칙 참고 | 웹서비스용 검색·근거 인용·대화 계층 신규 구축 |
| 경쟁률 | 3단 폴백 구조 활용 | 실제·지역 이력·통계 추정 구분 표시 |
| 일정·ICS | D-day와 ICS 로직 활용 | 확정 일정과 추정 일정 분리 |
| 변동 추적 | 스냅샷 diff 활용 | DB 기반 버전 이력과 변경 내역 표시 |
| 프로필·즐겨찾기 | 로컬 파일 스키마 참고 | 계정별 DB 저장과 동의·삭제 기능 신규 구축 |

## 공통 데이터 경계

### 공고

- 내부 ID와 기관 원본 ID
- 기관, 원문 URL, 원본 데이터 유형
- 지역, 세부 지역, 주거 유형, 공급 대상
- 모집 시작일, 종료일, 당첨 발표일, 계약 기간
- 접수중, 모집예정, 마감 상태
- 자격요건 요약과 구조화된 규칙
- 지원 또는 공급 내용, 세대수, 경쟁률
- 준비서류
- 수집 시각, 원문 갱신 시각, 내용 해시, 버전
- 실제값·추정값·확인 필요 여부를 나타내는 출처 메타데이터

### 사용자 프로필

- 연령과 거주 지역
- 가구 유형과 가구원
- 무주택 및 주택 보유 이력
- 소득·자산 구간
- 청약통장 가입기간과 납입횟수
- 혼인·자녀 등 특별공급 판단에 필요한 항목
- 관심 주거 유형과 희망 지역

민감 정보는 필요한 최소 범위만 저장하고, 사용 목적과 삭제 방법을 명확히 제공합니다.

### 자격 판정

- `eligible`: 현재 입력으로 충족
- `ineligible`: 명확한 조건 불충족
- `needs_confirmation`: 원문 또는 증빙 확인 필요
- 판정 근거와 사용한 규칙 버전
- 계산 시각과 사용한 공고 버전

화면에서는 이를 실제 자격 확정이 아닌 사전 점검 결과로 표시합니다.

## 기술적 보완이 필요한 부분

- 메모리 캐시는 서버 재시작에 취약하므로 필요한 데이터는 영속 저장소로 전환합니다.
- SH·GH처럼 일정이 없는 공고는 등록일 기반 추정만으로 접수중이라고 단정하지 않습니다.
- 경쟁률 통계 폴백과 예상 당첨·계약 일정은 추정값으로 표시합니다.
- HTML 크롤러는 기관 페이지 변경을 감지하는 테스트와 실패 알림이 필요합니다.
- 원문 추출 실패 시 빈 분석을 생성하지 않고 원문 직접 확인 상태로 전환합니다.
- 공고별 규칙은 적용일과 출처를 기록해 과거 판정 결과를 재현할 수 있어야 합니다.

## 도입 전 확인사항

k-apt-alert 포크와 원본 저장소 모두 GitHub 라이선스 메타데이터가 없고 루트 `LICENSE` 파일도 확인되지 않았습니다. 따라서 청나주 구현은 공개 API 필드와 관찰 가능한 동작을 기준으로 독립 작성했으며 원본 소스 코드를 복사하지 않습니다.
